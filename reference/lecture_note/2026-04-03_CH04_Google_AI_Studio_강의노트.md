## 이 챕터의 목적

- 참가자가 `AI가 답변을 잘하는 것`에서 한 걸음 더 나아가 `AI와 함께 실제 산출물을 만드는 방식`을 체감하게 한다.
- 핵심은 완성도 높은 제품 개발이 아니라, `짧은 지시 → 빠른 결과 확인 → 수정 요청 → 반복 개선`의 리듬을 익히게 하는 것이다.

- 중요한 것은 코드를 처음부터 다 짜는 것이 아니라, `의도와 수정 방향을 잘 설명하는 것`입니다.

## CH04-1 Google AI Studio 소개 및 접속 방법

### 강의 목표
- AI Studio가 무엇인지, 어디서 들어가고, 어떤 화면을 보게 되는지 익숙해지게 한다.
- 참가자가 `이걸로 실제 앱을 만들 수 있구나`라는 감각만 잡으면 충분하다.
- API 키나 세부 설정은 깊게 설명하지 말고, 오늘 실습에 필요한 최소한의 맥락만 준다.

### 강의 포인트
- AI Studio는 단순 채팅창이 아니라 `만들고, 고치고, 내려받고, 공유하는 작업대`다.
- `Playground`와 `Build`의 차이를 짧게만 설명한다.
- 오늘은 세부 기술보다 `어떻게 실습에 진입하는가`가 중요하다.
- 동시에, 결과물을 내 컴퓨터로 가져와 실행할 때는 왜 `API key`, `quota`, `환경 변수` 같은 요소가 따라오는지 감각만 잡게 한다.

### 진행 멘트 예시
- 여기서는 복잡한 설명보다 화면 감각만 익히겠습니다.
- Gemini를 대화형 도구로 썼다면, AI Studio는 그 능력을 앱 제작으로 확장하는 공간이라고 보면 됩니다.
- 오늘은 여기서 실제로 작은 앱을 만들고 수정해보는 것이 목적입니다.

### 참가자에게 보여줄 것
- 접속 경로
- Build 화면
- 프롬프트를 넣고 결과를 보는 위치
- 공유/다운로드가 가능한 흐름
- 결과물을 복사해 와도 내 PC에서 바로 안 될 수 있는 이유
- 개인 실험용 key와 조직 운영용 key를 같은 것으로 보면 안 된다는 점

### 최근 참고 영상
- `제미나이 3.1 제대로 쓰는 법! 구글 AI 스튜디오 플레이그라운드 활용 가이드`
	- 링크: `https://www.youtube.com/watch?v=irw2SOuu_UI`
	- 길이: 약 15분
	- 최근성: `1개월 전`
	- 용도: AI Studio가 단순 채팅창이 아니라 실험실이라는 감각을 잡게 한다.
- `말 한마디로 앱 만든다｜Google AI Studio Build로 생활 앱 3종 실습`
	- 링크: `https://www.youtube.com/watch?v=3IITukX8r8Q`
	- 길이: 약 14분
	- 최근성: `5개월 전`
	- 용도: Build에서 실제로 앱이 만들어지는 장면을 보기 좋다.

### 너무 길게 설명하지 않을 것
- quota
- billing
- 보안 정책
- API 구조 세부 설명

### 이 클립에서 꼭 한 번 짚을 문장
- AI Studio는 만들기의 출발점이고, API key는 실행의 출발점입니다.
- 결과물을 가져오는 것과 실행 환경까지 가져오는 것은 다른 문제입니다.
- 조직에서 앱을 돌릴 때는 기능보다 먼저 `키 관리와 운영 책임`이 따라옵니다.

## CH04-2 바이브 코딩이란

### 강의 목표
- 바이브 코딩을 `코드를 모르는 사람도 할 수 있다`로만 설명하지 말고, `요구사항을 점점 더 정확하게 다듬어 가는 협업 방식`으로 이해시키는 것이다.

### 강의 포인트
- 처음부터 완벽한 프롬프트를 쓰지 않아도 된다.
- 중요한 것은 첫 결과를 보고 수정 방향을 명확히 말하는 것이다.
- 좋은 바이브 코딩은 `아이디어 → 초안 → 검토 → 수정 → 재검토`의 반복이다.

### 진행 멘트 예시
- 바이브 코딩은 마법처럼 한 번에 끝나는 것이 아니라, 빠르게 초안을 보고 수정하는 작업입니다.
- 오히려 처음 결과가 조금 아쉬워도 괜찮습니다. 그다음 요청에서 방향을 잡는 것이 더 중요합니다.
- 결국 사람은 목적과 판단을 맡고, AI는 초안과 구현 속도를 맡는다고 생각하면 됩니다.

### 참가자에게 남겨야 할 감각
- 앱을 만드는 사람처럼 생각하되, 처음부터 개발자처럼 완벽하게 쓰려고 하지 않는다.
- 원하는 화면, 필요한 기능, 불편한 점을 말로 잘 설명하면 된다.

### 오프닝 영상 후보
- 메인 후보: `솔직히 바이브코딩 좋은듯`
	- 링크: `https://www.youtube.com/watch?v=tuH1ZTl9s9Q`
	- 길이: `4분 49초`
	- 채널: `코딩애플`
	- 이유: 개발 문화가 어떻게 바뀌는지, 왜 사람들이 바이브 코딩에 빠지는지 체감형으로 보여준다.
- 보조 후보: `"AI가 나보다 더 잘 짠다" (바이브 코딩)`
	- 링크: `https://www.youtube.com/watch?v=06cvEGFbIts`
	- 길이: `2분 57초`
	- 채널: `개발자방16`
	- 이유: AI가 초안을 쓰고 사람이 판단하는 역할 분담을 짧고 강하게 보여준다.
- 보조 후보: `바이브 코딩으로 가능 vs 불가능`
	- 링크: `https://www.youtube.com/shorts/OqK1CVIgePs`
	- 형식: `Shorts`
	- 이유: 수업 초반에 “어디까지는 잘 되고, 어디부터는 사람이 판단해야 하는지”를 가볍게 던지기 좋다.

### 영상 활용 팁
- 한 편만 쓸 때는 `코딩애플` 영상을 메인으로 쓰는 편이 가장 무난하다.
- 더 짧게 시작하고 싶으면 `개발자방16` 영상을 먼저 보여주고, 바로 실습으로 넘어가도 된다.
- Shorts는 메인 영상 대신 쓰기보다, 메인 영상 뒤에 한 번 더 감각을 압축하는 용도로 쓰는 편이 좋다.

### CH04-2 수업 운영 추천
- 메인 영상 1개만 쓰면 `코딩애플` 영상을 틀고 바로 실습으로 들어간다.
- 시간이 짧으면 `개발자방16` 영상을 대신 써도 된다.
- 영상 뒤에는 반드시 `이제 중요한 것은 코드를 다 외우는 것이 아니라, 원하는 결과를 더 잘 설명하고 고치는 것입니다`라는 문장으로 연결한다.

## CH04-3 바이브 코딩으로 웹앱 제작하기

### 강의 목표
- 실제로 짧은 앱 하나를 만들고, 수정 요청을 통해 결과가 발전하는 경험을 하게 한다.
- `시스템 개발 기술명세서 작성기` 같은 보조 도구를 활용해도 되고, 바로 시작 프롬프트를 넣어도 된다는 점을 보여준다.

### 현재 실습 흐름
- AI Studio Build로 이동
- 실제 Build 화면 이해
- Advanced settings에서 중요한 항목 5개 확인
- 시스템 개발 기술명세서 작성기 소개
- 대표 예제 3개로 시작
- 수정 요청으로 기능 고도화
- 각자 자기 앱을 만들고 Miro에 공유

### 강의 포인트
- 처음에는 단순한 예제로 시작해야 참가자들이 부담을 덜 느낀다.
- 결과를 바로 눈으로 볼 수 있는 예제가 좋다.
- 한 번 만들고 끝내지 말고, `기능 추가 요청`을 한 번 더 넣어 발전시키는 과정까지 보여줘야 한다.

### 추천 데모 리듬
- 1차: 가벼운 예제 하나 빠르게 생성
- 2차: 디자인 또는 기능 수정 요청
- 3차: 실무형 예제로 전환

### 추천 예제 활용 순서
- `Lotto Lucky Draw`
	- 웃긴 워밍업용
	- 빠르게 결과가 보여서 진입장벽이 가장 낮다
	- 버튼, 상태 저장, 애니메이션까지 짧은 시간 안에 확인할 수 있다
- `포켓몬 배틀 게임`
	- 재밌는 상호작용형 예제
	- 캐릭터 선택, HP 바, 턴 로직, 애니메이션이 한 번에 보여서 눈에 확 들어온다
	- 같은 프롬프트라도 디테일을 더 넣을수록 게임 완성도가 올라간다는 점을 체감하기 좋다
- `Executive Briefing Formatter`
	- 가장 진지한 실무형 예제
	- CH02~CH03에서 만들었던 회의 메모, 리서치 결과, 브리핑 흐름과 직접 연결된다
	- 수업 대상인 임원/리더와 가장 바로 연결되는 도구다

### 진행 멘트 예시
- 이제부터는 결과를 보고 바로 고쳐나가는 흐름을 보겠습니다.
- 중요한 것은 한 번에 완벽한 앱이 아니라, 짧은 수정 요청으로 점점 좋아지는 경험입니다.
- 같은 앱도 어떤 요구사항을 먼저 주느냐에 따라 결과가 달라진다는 점을 느껴보시면 됩니다.

### 개인 실습과 공유 흐름
- 세 예제는 따라 하기용 출발점일 뿐이라고 먼저 말해준다.
- 참가자는 세 예제 중 하나를 변형해도 되고, 자기 업무와 연결된 새로운 앱을 만들어도 된다고 안내한다.
- 실습 마지막에는 `앱 링크`, `화면 캡처`, `핵심 프롬프트` 중 하나 이상을 Miro에 올리게 한다.
- Miro에서는 결과물 자체보다도 `어떤 목적을 넣었고`, `어떤 수정 요청을 했고`, `무엇이 달라졌는지`를 같이 보게 한다.
- 강사는 2~3개의 결과만 골라 함께 보며, 같은 도구로도 결과가 얼마나 다르게 나오는지 짚어주면 된다.

## 강의 운영 팁

### 시간 배분 제안
- AI Studio 소개: 5분
- 바이브 코딩 개념 설명: 5분
- 첫 앱 생성 데모: 10분
- 수정 요청과 고도화: 10분
- 참가자 실습 및 공유: 10~15분

### 참가자가 막힐 때 개입하는 방법
- 무엇을 만들고 싶은지 한 줄로 먼저 말하게 한다.
- 사용자, 목적, 핵심 기능 3가지만 먼저 정하게 한다.
- 결과가 이상하면 `어디가 불편한지`를 말하게 한다.
- UI보다 기능부터, 기능보다 흐름부터 잡게 한다.

### 강사가 반복해서 강조할 문장
- 처음부터 완벽하게 쓰려고 하지 않아도 됩니다.
- 원하는 결과를 본 다음, 수정 방향을 더 잘 말하는 것이 중요합니다.
- 작은 앱 하나를 끝까지 고쳐보는 것이 훨씬 더 많은 것을 남깁니다.

## 챕터 마무리 멘트
- 지금까지는 AI를 읽기와 정리에 썼다면, 이제는 만드는 일에도 붙일 수 있다는 것을 봤습니다.
- 오늘 중요한 것은 도구를 외우는 것이 아니라, `아이디어를 빠르게 앱으로 바꾸고 계속 수정해 나가는 감각`입니다.
- 이후 Hi-D Code 시연에서는 이런 흐름이 사내 환경에서는 어떻게 이어질 수 있는지 보게 됩니다.

## 현재 기준 정리 포인트
- CH04의 실제 visible 흐름은 `Google AI Studio 소개 → 바이브 코딩이란 → 웹앱 제작 실습`으로 이해하면 된다.
- 첫 클립은 너무 기술적으로 깊어지지 않게 유지하는 것이 좋다.
- 세 번째 클립은 예제를 `웃긴 것 / 재밌는 것 / 진지한 것` 3개로 압축해 진행하는 편이 가장 자연스럽다.
- 실습 후에는 참가자 결과를 Miro에 모아 함께 보는 시간이 반드시 필요하다.
